# Vulnerability Assessment Report: CyberBank IDOR Analysis
**Project:** Web Application Security - IDOR  

---

## 1. Introduction
This report documents the security assessment of the **CyberBank** web application (`web0x06.hbtn`). The objective was to identify and exploit vulnerabilities related to **Insecure Direct Object Reference (IDOR)** within the user profile and payment processing modules.

## 2. Methodology
The assessment was performed using manual testing via Browser Developer Tools.
* **Tools used:** Firefox/Chrome DevTools (Network, Console, Debugger).
* **Techniques:** API request manipulation, session token analysis, and parameter tampering to bypass business logic.

---

## 3. Vulnerability Details

### 3.1 Unauthorized Data Exposure (IDOR) - Flag 2
* **Description:** The API endpoint `/api/customer/info/me` (and variations like `/info/[id]`) fails to validate if the requester owns the account data being returned.
* **Impact:** **Critical.** An attacker can access sensitive financial data (balances, income, expenses) and personal information of any user by simply knowing their UUID.
* **Proof of Concept:**
    ```javascript
    fetch('/api/customer/info/me').then(r => r.json()).then(console.log);
    ```
* **Evidence:** **Flag 2:** `b98aa9a064f5a21846ceb6edfc31c3aa`

### 3.2 Payment Logic Manipulation (Transaction Hijacking) - Flag 3
* **Description:** A two-stage IDOR vulnerability exists in the payment workflow. The system allows a user to initiate a payment for a different account (`victimId`) and then confirm it using their own `accountId`.
* **Impact:** **High.** This allows unauthorized plan upgrades or credit card fraud by misattributing transactions.
* **Reproduction Steps:**
    1. Intercept `POST /api/cards/init_payment` and change `account_id` to the victim's ID.
    2. Retrieve the `transaction_id` and `OTP` from the server response (Leaked sensitive data).
    3. Send `POST /api/cards/confirm_payment` using the attacker's `account_id` with the intercepted `transaction_id`.
* **Evidence:** **Flag 3:** `e9d5b76927c2ec02fef1dd18af22ccb3`

---

## 4. Additional Security Flaws
* **Sensitive Data Leakage:** The API returns the `OTP` (One-Time Password) and `CVV` in the response body during the initiation phase, making the 2FA process completely redundant.
* **Insecure API Routing:** Multiple redundant endpoints (e.g., `/api/me`, `/api/customer/info/me`) exist with varying levels of security, increasing the attack surface.

## 5. Recommendations
1.  **Implement Server-Side Authorization:** Every API request must verify that the authenticated session matches the `account_id` in the request body/parameters.
2.  **Use Indirect References:** Replace predictable internal IDs with non-enumerable, temporary tokens for transaction processing.
3.  **Secure OTP Delivery:** OTPs must never be returned in an API response; they should be sent via out-of-band channels (SMS/Email).

## 6. Conclusion
The CyberBank application exhibits severe IDOR vulnerabilities that compromise both user privacy and financial integrity. Implementing robust server-side validation and consistent access control across all API routes is mandatory for remediation.

## Introduction
In the realm of cybersecurity, few vulnerabilities have stood the test of time quite like the **Buffer Overflow**. Despite being known for decades, it remains one of the most critical security flaws in software development. 

A buffer overflow occurs when a program attempts to write more data to a fixed-size block of memory (a buffer) than it is allocated to hold. The excess data spills over into adjacent memory locations, corrupting valid data, crashing the system, or worse—allowing attackers to execute arbitrary malicious code. 

The consequences of these attacks are severe:
* **System Crashes:** Overwriting critical memory can lead to Segmentation Faults and Denial of Service (DoS).
* **Privilege Escalation:** Attackers can manipulate program execution to gain administrative access.
* **Data Theft:** Sensitive information resting in memory can be accessed or altered.

## The Mechanics: How Buffer Overflows Occur
To understand buffer overflows, we must look at how computer memory manages running programs, specifically the **Stack**. 

When a function is called in languages like C or C++, a "stack frame" is created. It typically contains:
1. **Local Variables:** Fixed-size buffers allocated for temporary data.
2. **The Saved Frame Pointer (EBP/RBP):** Used to reference local variables.
3. **The Return Address:** A pointer telling the CPU where to go next after the function finishes.

Memory fills from lower addresses to higher addresses. If a program accepts user input and writes it into a buffer without checking its length (using unsafe functions like `strcpy` or `gets`), the input will fill the buffer. If the input is too long, it continues writing upward, directly overwriting the frame pointer and eventually the **Return Address**.

By corrupting this return address, an attacker can redirect the CPU to execute a different set of instructions stored elsewhere in memory (like injected Shellcode).
